# Spec-Driven Development (SDD) Workflow
## VibeLabs "바이브 코딩" 방법론 적용 가이드

본 프로젝트는 **Spec-Driven Development (스펙 주도 개발)** 방식을 따릅니다.
모든 개발 작업은 코드를 작성하기 전에 **문서(Spec)를 먼저 정의**하는 것부터 시작합니다.

### 1. SDD 프로세스 (The Workflow)

**Step 1: 요구사항 정의 (Define Specs)**
*   작업 전, 관련 문서를 먼저 업데이트합니다.
    *   **기능 정의:** `PRD.md` 에 기능의 목적, 사용자 흐름, 예외 처리를 기록.
    *   **데이터 설계:** `DATABASE_DESIGN.md` 에 데이터 구조 변경사항 기록.
    *   **기술 설계:** `TRD.md` 에 사용할 라이브러리, 알고리즘, 파일 구조 기록.

**Step 2: 승인 및 리뷰 (Review)**
*   작성된 Spec에 대해 사용자(Human) 또는 AI의 검토를 거칩니다.
*   "이대로 만들면 문제없나?"를 문서 단계에서 확인하여 재작업 비용을 줄입니다.

**Step 3: 구현 (Implementation)**
*   확정된 Spec을 바탕으로 코드를 작성합니다.
*   AI에게 프롬프트를 줄 때도 *"PRD의 2.1항목을 구현해줘"*와 같이 명확한 지시가 가능해집니다.

**Step 4: 검증 (Verification)**
*   구현된 결과물이 Spec과 일치하는지 확인합니다.

---

### 2. 문서 유지보수 규칙
*   코드가 변경되면 반드시 관련 문서도 함께 업데이트되어야 합니다. (Sync Code & Docs)
*   문서는 '죽은 문서'가 아닌, 개발의 '지도(Map)' 역할을 해야 합니다.
