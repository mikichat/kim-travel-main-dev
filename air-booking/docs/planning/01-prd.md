## 1. 개요

### 핵심 설계 원칙: 원데이터 하나 → 두 가지 뷰

이 프로젝트의 가장 중요한 원칙은 다음과 같습니다:

```
🎯 DB에는 PNR 원본 데이터 1개만 저장.
   이 하나의 데이터를 두 가지 화면으로 보여줍니다:

1️⃣  고객용 뷰 (항공편 변환기)
   • 대상: 일반 고객 (항공 업무를 모름)
   • 표시: 도시명(인천), 읽기 쉬운 날짜, 이미지/랜딩카드
   • 목적: 카톡/메일로 고객에게 전달

2️⃣  예약장부용 뷰 (air-booking)
   • 대상: 여행사 직원, 대리점 관계자
   • 표시: 공항코드(ICN), 마감일(NMTL/TL), 정산, 발권
   • 목적: 내부 예약관리

📌 데이터는 하나, 보여주는 방식만 다르다.
   같은 PNR을 두 곳에 따로 입력하는 것은 절대 금지.
```

### 1.1 문제 상황

현재 시스템의 문제점:

1. **데이터 중복 저장**: 동일한 PNR이 3곳에 저장됨
   - flight_saves (변환기 localStorage)
   - flight_schedules (항공편 관리)
   - air_bookings (예약장부)

2. **PC 간 데이터 불일치**: localStorage 의존으로 PC마다 다른 데이터 표시

3. **동기화 실패**: 한쪽에서 수정해도 다른 쪽에 반영 안됨

4. **중복 입력**: 직원이 같은 PNR을 두 곳에 따로 입력해야 함

5. **UI 불일치**: 변환기와 air-booking 화면 분리로 혼란

### 1.2 해결 방안

> **한 가지 입력 → 양쪽 모두 자동 표시**

1. **통합 서버**: main(포트 5001, JS) + air-booking(포트 5174, TS) → 하나의 Express+TypeScript 서버
2. **통합 DB**: 3개 테이블 → 1개 통합 테이블 (flight_bookings)
3. **실시간 동기화**: 한쪽 입력 → 즉시 DB 저장 → 양쪽 화면 갱신
4. **localhost 통일**: 양쪽 모두 하나의 서버에서 제공

### 1.3 성공 기준

- [ ] PNR 한 번 입력 → 양쪽에서 동시 표시
- [ ] 어떤 PC에서든 같은 데이터 보임 (localStorage 제거)
- [ ] 기존 기능 100% 유지 (하위 호환)
- [ ] 중복 데이터 제거 (마이그레이션)
- [ ] 응답 시간 500ms 이내

## 2. 사용자 및 사용 시나리오

### 2.1 사용자 유형

| 역할 | 설명 | 사용 기능 |
|------|------|---------|
| 여행사 직원 | 예약, 이미지 생성, 고객 전달 | air-booking + 변환기 |
| 시스템 관리자 | 항공편 스케줄 관리 | flight-schedule.html |

### 2.2 시나리오 A: air-booking → 변환기

```
1. 직원이 air-booking에서 예약장부에 PNR 입력
   (고객명, 항공편, 탑승객 정보 등)

2. DB 저장 (flight_bookings 테이블)

3. 변환기 '저장된 항공편' 탭에 자동 표시
   - PNR, 항공편, 탑승객 목록

4. 이미지/랜딩카드 자동 생성
   (기존 기능)

5. 고객에게 카톡/메일로 전달
```

### 2.3 시나리오 B: 변환기 → air-booking

```
1. 직원이 변환기에서 PNR 텍스트 붙여넣기

2. 변환기가 파싱하여 저장 버튼 클릭

3. DB 저장 (flight_bookings 테이블)

4. air-booking 예약장부에 자동 표시
   - NMTL/TL 마감, 정산, 발권 관리 가능

5. 직원이 예약 상태 수정/삭제 가능
```

### 2.4 시나리오 C: 항공편 관리

```
1. 관리자가 항공편 스케줄 CRUD
   (항공사, 편명, 출발/도착 시간 등)

2. DB 저장 (flight_schedules 테이블)

3. 예약 입력 시 항공편 검색/선택
   (자동완성)

4. 모든 예약/저장기록에서 해당 항공편 통계
```

## 4. 기술 요구사항

### 4.1 서버 아키텍처

- **통합 서버**: Express + TypeScript
- **포트**: 기존 main(5001) 또는 새 포트로 통일
- **API**: RESTful (향후 GraphQL 검토)

### 4.2 데이터베이스

- **DB**: SQLite (travel_agency.db)
- **테이블**: flight_bookings (통합 테이블) + flight_schedules
- **마이그레이션**: 기존 데이터 100% 이관
- **WAL 모드**: 동시 접근 안정화

### 4.3 프론트엔드

- **air-booking**: React (기존 유지)
- **변환기**: vanilla HTML/JS (통합)
- **CDN**: 로컬 import로 변경 (CORS 회피)

### 4.4 보안

- **인증**: 로컬호스트 (기존 유지, 향후 개선)
- **CORS**: 내부망 (5001, 5174 통합)
- **클립보드**: HTTPS 또는 디바이스 공유 (다음 버전)

### 4.5 성능

- **응답 시간**: 500ms 이내
- **동시 사용자**: 5명 (여행사 직원)
- **DB 용량**: 1년 치 데이터 < 500MB

## 5. 제약 조건 & 가정

### 5.1 제약 조건

1. **기존 기능 100% 유지**: 새로운 기능 추가 전에 마이그레이션 완료 필수
2. **여행사 내부망**: 외부 접근 불가 (HTTPS 불필요)
3. **SQLite 용량**: 다중 접근 시 WAL 모드 필수
4. **PC 간 동기화**: localhost 문제로 온라인/오프라인 모드 불가

### 5.2 가정

1. 직원은 동일한 WiFi 또는 LAN에 연결
2. 서버는 여행사 사무실의 특정 PC에서 24/7 실행
3. 중복 PNR은 최신 데이터 기준으로 유지
4. 삭제된 예약은 복구 불가 (아카이브 기능은 향후)
