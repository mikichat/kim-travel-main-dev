# 03-user-flow.md — 사용자 흐름도 (시나리오별)

**프로젝트명**: DB 통합 및 서버 일원화
**버전**: 1.0
**작성일**: 2026-04-01

---

## 개요

**핵심 설계 원칙: 원데이터 하나 → 두 가지 뷰**

DB에는 PNR 원본 데이터 1개만 저장하고, 이를 두 가지 화면으로 표시합니다:
- 고객용 뷰 (항공편 변환기): 도시명, 이미지/랜딩카드
- 예약장부용 뷰 (air-booking): 공항코드, 마감일, 정산, 발권

같은 PNR을 두 곳에 따로 입력하는 것은 절대 금지입니다.

---

이 문서는 실제 사용 시나리오에 따른 사용자의 조작 흐름을 보여줍니다.

**사용자**: 여행사 직원 (동일 직원이 air-booking + 변환기 모두 사용)

---

## 시나리오 1: Air-Booking에서 PNR 등록 후 고객에게 전달

### 흐름도

```
┌─────────────────────────────────────────┐
│ 여행사 직원 @ air-booking               │
└─────────────────────────────────────────┘
              │
              ↓
    ┌─────────────────────────┐
    │ 예약장부 탭 열기          │
    │ (현재 예약 목록 조회)      │
    └─────────────────────────┘
              │
              ↓
    ┌─────────────────────────────┐
    │ "+ 새 예약" 버튼 클릭        │
    │ (입력 폼 열기)              │
    └─────────────────────────────┘
              │
              ↓
    ┌─────────────────────────────────────────────┐
    │ 예약 정보 입력:                             │
    │ - PNR: ABC123                             │
    │ - 항공편: KE 1001 (ICN→NRT, 4월 15일)      │
    │ - 탑승객: 김철수(M12345678), 1A석           │
    │ - 마감: NMTL 4월 10일, TL 4월 13일          │
    │ - 상태: 예약 (booked)                      │
    └─────────────────────────────────────────────┘
              │
              ↓
    ┌─────────────────────────┐
    │ "저장" 버튼 클릭        │
    └─────────────────────────┘
              │
              ↓ (네트워크)
    ┌────────────────────────────────┐
    │ POST /api/bookings              │
    │ {                              │
    │   pnr: "ABC123",               │
    │   airline: "KE",               │
    │   flight_number: "1001",       │
    │   ...                          │
    │ }                              │
    └────────────────────────────────┘
              │
              ↓ (DB 저장)
    ┌────────────────────────────────┐
    │ flight_bookings 테이블에 저장   │
    │ id=1, source='booking'         │
    └────────────────────────────────┘
              │
              ↓
    ┌──────────────────────────────┐
    │ 예약장부에 목록 갱신 표시     │
    │ (저장 성공 메시지)            │
    └──────────────────────────────┘
              │
              ↓
    ┌─────────────────────────────┐
    │ (옵션) 발권/요금증명서 생성   │
    │        (기존 기능 유지)      │
    └─────────────────────────────┘
              │
              ↓
    ┌─────────────────────────────────────────┐
    │ 예약장부에서 "고객전달" 버튼 클릭       │
    │ (이미지/랜딩카드 생성 & 다운로드)       │
    └─────────────────────────────────────────┘
              │
              ↓
    ┌─────────────────────────────────────────┐
    │ 생성된 이미지를 카톡/메일로 고객 전달   │
    └─────────────────────────────────────────┘
```

### 주요 상호작용

| 단계 | 사용자 동작 | 시스템 반응 |
|------|-----------|-----------|
| 1 | 예약장부 탭 열기 | GET /api/bookings → 목록 표시 |
| 2 | "+ 새 예약" 클릭 | 예약 입력 폼 열기 |
| 3 | 정보 입력 | 폼 유효성 검사 (실시간) |
| 4 | "저장" 클릭 | POST /api/bookings → DB 저장 |
| 5 | 저장 완료 | 목록 갱신, 성공 메시지 |
| 6 | "고객전달" 클릭 | 이미지 생성 & 다운로드 |

### 예상 소요 시간

- 예약 입력: 2~3분
- DB 저장: 500ms 이내
- 이미지 생성: 1~2초
- **총 소요 시간**: 3~5분

## 시나리오 2: 변환기에서 PNR 입력 → air-booking 자동 표시

### 흐름도

```
┌─────────────────────────────────────────┐
│ 여행사 직원 @ 변환기 (standalone HTML)  │
└─────────────────────────────────────────┘
              │
              ↓
    ┌──────────────────────────────┐
    │ PNR 텍스트 붙여넣기           │
    │ (예: "KE 1001 ICN NRT 4/15") │
    └──────────────────────────────┘
              │
              ↓
    ┌───────────────────────────────┐
    │ "변환" 버튼 클릭              │
    │ (텍스트 파싱)                 │
    └───────────────────────────────┘
              │
              ↓
    ┌────────────────────────────────────┐
    │ 파싱 결과 표시:                    │
    │ - 항공편: KE 1001                │
    │ - 경로: ICN → NRT                │
    │ - 날짜: 4월 15일                  │
    │ - 탑승객: [입력 필요]              │
    └────────────────────────────────────┘
              │
              ↓
    ┌─────────────────────────┐
    │ "저장" 버튼 클릭        │
    └─────────────────────────┘
              │
              ↓ (네트워크)
    ┌────────────────────────────────┐
    │ POST /api/bookings              │
    │ {                              │
    │   pnr: "ABC123",               │
    │   source: "converter",         │
    │   ...                          │
    │ }                              │
    └────────────────────────────────┘
              │
              ↓ (DB 저장)
    ┌────────────────────────────────┐
    │ flight_bookings 테이블에 저장   │
    │ id=2, source='converter'       │
    └────────────────────────────────┘
              │
              ↓
    ┌──────────────────────────────┐
    │ 변환기에서 저장 성공 메시지   │
    │ (목록 갱신)                   │
    └──────────────────────────────┘
              │
              ↓ (자동 동기화)
    ┌──────────────────────────────────────────┐
    │ Air-booking (다른 탭/창 열림)             │
    │ → Polling: GET /api/bookings             │
    │ → 새로운 예약 감지                       │
    │ → 예약장부 목록에 자동 추가               │
    │ (새로운 행으로 표시, 배경색 강조)         │
    └──────────────────────────────────────────┘
              │
              ↓
    ┌─────────────────────────────────────┐
    │ 여행사 직원이 air-booking 열어서    │
    │ 새로 추가된 예약을 확인             │
    │ (PNR, 항공편, 탑승객 등)            │
    └─────────────────────────────────────┘
              │
              ↓
    ┌─────────────────────────────────────┐
    │ (옵션) 마감 날짜 입력 및 상태 수정  │
    │ → PUT /api/bookings/:id             │
    └─────────────────────────────────────┘
```

### 주요 상호작용

| 단계 | 사용자 동작 | 시스템 반응 |
|------|-----------|-----------|
| 1 | PNR 텍스트 입력 | - |
| 2 | "변환" 클릭 | 텍스트 파싱 및 표시 |
| 3 | "저장" 클릭 | POST /api/bookings → DB 저장 |
| 4 | - | 변환기: 저장 성공 메시지 |
| 5 | air-booking 탭 / 새 창 열기 | Polling: GET /api/bookings |
| 6 | - | 새 예약 자동 추가 표시 |

### 예상 소요 시간

- PNR 입력 & 파싱: 1~2초
- DB 저장: 500ms 이내
- Polling 갱신: 1초 (최대 1초)
- **총 소요 시간**: 2~3초 (실시간 동기화)

## 시나리오 3: air-booking에서 예약 수정 → 변환기 자동 갱신

### 흐름도

```
┌──────────────────────────────────┐
│ 여행사 직원 @ air-booking        │
│ (예약장부, 기존 예약 선택됨)     │
└──────────────────────────────────┘
              │
              ↓
    ┌─────────────────────────────┐
    │ 예약 행 클릭 (펼쳐보기)      │
    │ (예약 상세 보기)             │
    └─────────────────────────────┘
              │
              ↓
    ┌────────────────────────────────────┐
    │ 수정 정보:                         │
    │ - 마감 상태: "NMTL" → "TL"        │
    │ - 발권 여부: false → true         │
    │ - 비고: 텍스트 추가               │
    └────────────────────────────────────┘
              │
              ↓
    ┌─────────────────────────────┐
    │ "저장" 버튼 클릭            │
    └─────────────────────────────┘
              │
              ↓ (네트워크)
    ┌────────────────────────────────┐
    │ PUT /api/bookings/:id           │
    │ {                              │
    │   status: "ticketed",          │
    │   remarks: "발권 완료",         │
    │   ...                          │
    │ }                              │
    └────────────────────────────────┘
              │
              ↓ (DB 수정)
    ┌────────────────────────────────┐
    │ flight_bookings 테이블 UPDATE  │
    │ id=1, updated_at 갱신          │
    └────────────────────────────────┘
              │
              ↓
    ┌──────────────────────────────┐
    │ air-booking: 수정 성공 메시지 │
    │ (목록 행 색상 변경 등)        │
    └──────────────────────────────┘
              │
              ↓ (자동 동기화)
    ┌──────────────────────────────────────────┐
    │ 변환기 (다른 탭/창 열림)                  │
    │ → Polling: GET /api/bookings             │
    │ → 기존 예약 데이터 변경 감지              │
    │ → "저장된 항공편" 목록의 해당 행 갱신     │
    │ (상태 표시 업데이트)                     │
    └──────────────────────────────────────────┘
              │
              ↓
    ┌──────────────────────────────┐
    │ 변환기: 발권 완료 표시        │
    │ (예약장부에서도 반영됨)       │
    └──────────────────────────────┘
```

### 주요 상호작용

| 단계 | 사용자 동작 | 시스템 반응 |
|------|-----------|-----------|
| 1 | 예약 행 클릭 | 예약 상세 표시 |
| 2 | 정보 수정 | 폼 유효성 검사 (실시간) |
| 3 | "저장" 클릭 | PUT /api/bookings/:id → DB 수정 |
| 4 | - | air-booking: 성공 메시지 & 목록 갱신 |
| 5 | - | 변환기: Polling으로 갱신 감지 |
| 6 | - | 변환기: 목록 행 자동 갱신 |

### 예상 소요 시간

- 예약 상세 보기: 1초
- 정보 수정: 1~2분
- DB 저장: 500ms 이내
- Polling 갱신: 1초 (최대 1초)
- **총 소요 시간**: 2~3분

## 시나리오 4: 항공편 검색 & 자동완성

### 흐름도

```
┌────────────────────────────────────┐
│ 여행사 직원 @ air-booking 또는 변환기 │
│ (PNR 입력 중)                      │
└────────────────────────────────────┘
              │
              ↓
    ┌────────────────────────────┐
    │ "항공편 선택" 드롭다운 클릭 │
    └────────────────────────────┘
              │
              ↓
    ┌────────────────────────────┐
    │ "KE" 입력 (항공사 코드)     │
    └────────────────────────────┘
              │
              ↓ (네트워크)
    ┌────────────────────────────────────────┐
    │ GET /api/schedules/search?airline=KE   │
    └────────────────────────────────────────┘
              │
              ↓ (DB 검색)
    ┌────────────────────────────────────────┐
    │ flight_schedules에서 매칭 항공편 검색  │
    │ (airline='KE' WHERE departed_date > now)│
    └────────────────────────────────────────┘
              │
              ↓
    ┌─────────────────────────────────────┐
    │ 결과 표시 (드롭다운):               │
    │ - KE 1001 ICN NRT 4/15              │
    │ - KE 1002 ICN NRT 4/16              │
    │ - KE 1003 ICN NRT 4/17              │
    │ - ...                               │
    └─────────────────────────────────────┘
              │
              ↓
    ┌──────────────────────────────────┐
    │ 사용자가 "KE 1001 ICN NRT 4/15" 선택 │
    │ (폼 필드 자동완성)                │
    └──────────────────────────────────┘
              │
              ↓
    ┌───────────────────────────────┐
    │ airline="KE"                  │
    │ flight_number="1001"          │
    │ route_from="ICN"              │
    │ route_to="NRT"                │
    │ departure_date="2026-04-15"   │
    └───────────────────────────────┘
```

### 주요 상호작용

| 단계 | 사용자 동작 | 시스템 반응 |
|------|-----------|-----------|
| 1 | 드롭다운 클릭 | 검색 입력 필드 활성화 |
| 2 | 항공사 코드 입력 | GET /api/schedules/search |
| 3 | - | 매칭된 항공편 목록 표시 |
| 4 | 항공편 선택 | 폼 필드 자동완성 |

### 예상 소요 시간

- 드롭다운 열기: 즉시
- 검색 & 표시: 500ms 이내
- **총 소요 시간**: 1초 이내

## 시나리오 6: PNR 메모에서 원본 꺼내 재변환

### 흐름도

```
┌──────────────────────────────────┐
│ 여행사 직원 @ 변환기              │
│ (저장된 항공편 목록에서 항공편 클릭) │
└──────────────────────────────────┘
              │
              ↓
    ┌──────────────────────────────┐
    │ 항공편 상세 정보 표시         │
    │ - PNR: ABC123               │
    │ - 항공편: KE 1001           │
    │ - 메모: (원본 PNR 텍스트 표시) │
    │   "KE 1001 ICN NRT 4/15..." │
    └──────────────────────────────┘
              │
              ↓
    ┌──────────────────────────────┐
    │ "수정" 버튼 클릭             │
    └──────────────────────────────┘
              │
              ↓
    ┌──────────────────────────────────────┐
    │ 메모에서 원본 PNR 추출               │
    │ (텍스트 파싱 UI에 자동 삽입)         │
    └──────────────────────────────────────┘
              │
              ↓
    ┌──────────────────────────────────────┐
    │ "재변환" 버튼 클릭                   │
    │ (메모의 원본 PNR 재파싱)             │
    └──────────────────────────────────────┘
              │
              ↓ (네트워크)
    ┌────────────────────────────────┐
    │ POST /api/converter/parse       │
    │ {                              │
    │   text: "KE 1001 ICN NRT 4/15" │
    │ }                              │
    └────────────────────────────────┘
              │
              ↓
    ┌────────────────────────────────┐
    │ 파싱 결과 표시 (수정 가능)      │
    │ - 항공편, 경로, 날짜, 탑승객   │
    └────────────────────────────────┘
              │
              ↓
    ┌──────────────────────────────┐
    │ "저장" 버튼 클릭             │
    │ (PUT /api/bookings/:id)      │
    └──────────────────────────────┘
              │
              ↓ (DB 수정)
    ┌────────────────────────────────┐
    │ flight_bookings 테이블 UPDATE  │
    │ - 수정된 항공편 정보           │
    │ - 메모에 새로운 원본 저장      │
    └────────────────────────────────┘
              │
              ↓
    ┌──────────────────────────────┐
    │ 수정 완료 메시지              │
    │ (변환기 & 예약장부 동시 갱신) │
    └──────────────────────────────┘
```

### 주요 상호작용

| 단계 | 사용자 동작 | 시스템 반응 |
|------|-----------|-----------|
| 1 | 저장된 항공편 클릭 | 상세 정보 표시 (메모 포함) |
| 2 | "수정" 클릭 | 메모에서 원본 PNR 추출 |
| 3 | (선택) 텍스트 편집 | - |
| 4 | "재변환" 클릭 | POST /api/converter/parse |
| 5 | - | 파싱 결과 표시 (수정 가능) |
| 6 | "저장" 클릭 | PUT /api/bookings/:id → DB 수정 |
| 7 | - | 수정 성공 & 양쪽 목록 갱신 |

### 예상 소요 시간

- 메모에서 원본 추출: 즉시
- 재파싱: 500ms 이내
- 수정 & 저장: 500ms 이내
- **총 소요 시간**: 1초 이내 (실시간)

## 시나리오 8: 컴퓨터 재시작 후 데이터 확인

### 흐름도

```
┌───────────────────────────────────┐
│ 여행사 직원 @ PC 재시작           │
│ (기존 air-booking / 변환기 탭 닫힘) │
└───────────────────────────────────┘
              │
              ↓
    ┌───────────────────────────────┐
    │ air-booking 다시 열기         │
    │ (포트 5174 또는 새 서버 포트) │
    └───────────────────────────────┘
              │
              ↓ (네트워크)
    ┌────────────────────────────────┐
    │ GET /api/bookings              │
    │ (localStorage 없음 → DB 쿼리)  │
    └────────────────────────────────┘
              │
              ↓ (DB 조회)
    ┌──────────────────────────────────┐
    │ flight_bookings 테이블에서 전체 조회 │
    │ (모든 예약이 그대로 존재)         │
    └──────────────────────────────────┘
              │
              ↓
    ┌──────────────────────────────┐
    │ 예약장부: PC 재시작 전 데이터 │
    │ 모두 표시됨 (손실 없음)      │
    └──────────────────────────────┘
              │
              ↓
    ┌───────────────────────────────┐
    │ 변환기도 다시 열기             │
    │ (동일한 데이터 표시)          │
    └───────────────────────────────┘
              │
              ↓
    ┌───────────────────────────────────────┐
    │ 양쪽 데이터 일치:                     │
    │ - 같은 PNR 목록                      │
    │ - 같은 항공편 정보                    │
    │ - 같은 마감 상태                      │
    │ (localStorage 의존 제거됨)           │
    └───────────────────────────────────────┘
```

### 주요 차이점

| 항목 | 기존 (localStorage) | 개선 (서버 DB) |
|------|-------------------|----------------|
| PC 재시작 후 | 재시작 전 탭에만 데이터 | 모든 데이터 복구 |
| 다른 PC | 데이터 없음 | 같은 데이터 표시 |
| 브라우저 캐시 삭제 | 데이터 손실 | 데이터 유지 |
